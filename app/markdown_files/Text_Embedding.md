## 一、数据入库流程（上传文件）

### Step 1: 保存原始文件

```
用户在 Web 界面上传 xlsx 文件（支持多文件）
    │
    ▼
xlsx 文件保存到 report_template/ 目录
（自动跳过 ~$ 开头的临时文件和非 xlsx 文件）
```

**对应代码：** `web.py` → `upload_and_process()`

**示例：**

```
上传: 绥化_报告_模版2.xlsx
保存: report_template/绥化_报告_模版2.xlsx
```

### Step 2: 按行切片

```
读取 xlsx 第一行作为表头
每一行数据 → 一个 Markdown 文件
保存到 xlsx_slices/ 目录
```

**切片后的 md 文件格式：**

```markdown
## 绥化*报告*模版2

| 检查类型 | 部位 | 检查项目 | 诊断结论 | 影像学表现                    | 影像学意见         | 来源up_id | 来源id  | 模态/部位/检查       |
| -------- | ---- | -------- | -------- | ----------------------------- | ------------------ | --------- | ------- | -------------------- |
| CT       | 头颅 | 颅脑平扫 | 脑出血   | 侧基底节区见团块状高密度影... | 侧基底节区脑出血。 | 3000017   | 3000073 | CT / 头颅 / 颅脑平扫 |
```

**命名规则：** `{xlsx文件名}_row{行号}.md`，例如 `绥化_报告_模版2_row1.md`

**对应代码：** `web.py` → `_xlsx_to_slices()` + `_slice_to_md()`

### Step 3: 增量检查

```
查询 Milvus 已有的 source 集合
对比新切片的文件名
过滤掉已入库的切片，只处理新增的
```

**为什么需要增量检查？** 避免重复向量化，节省 API 调用和时间。同一个 xlsx 多次上传不会产生重复数据。

**对应代码：** `web.py` → `upload_and_process()` 中的增量检查逻辑

### Step 4: 解析 md → 自然语言文本

```
提取 md 文件中的表头和数据行
将每个字段拼接为自然语言文本
同时提取元数据
```

**拼接规则：** `字段名：值`，用换行符连接

**拼接后的文本（用于向量化）：**

```
检查类型：CT
部位：头颅
检查项目：颅脑平扫
诊断结论：脑出血
影像学表现：侧基底节区见团块状高密度影，CT值约61HU，大小约2.8cm×3.0cm，灶周见低密度水肿带，侧侧脑室受压，三脑室受压变形，中线结构向对侧移位。脑干及小脑未见异常。
影像学意见：侧基底节区脑出血。
来源up_id：3000017
来源id：3000073
模态/部位/检查：CT / 头颅 / 颅脑平扫
```

**元数据（单独存储，支持过滤查询）：**

| 字段     | 值       |
| -------- | -------- |
| 检查类型 | CT       |
| 部位     | 头颅     |
| 检查项目 | 颅脑平扫 |
| 诊断结论 | 脑出血   |

**对应代码：** `web.py` → `_parse_md_slice()`

### Step 5: 批量向量化

```
调用 bge-m3 Embedding API
每 16 条为一批次
失败自动重试 5 次（间隔递增：5s, 10s, 15s, 20s）
每条文本 → 1024 维向量
```

**API 请求格式：**

```json
{
  "model": "bge-m3",
  "input": ["检查类型：CT\n部位：头颅\n...", "检查类型：MRI\n..."]
}
```

**对应代码：** `web.py` → `_get_embeddings_batch()`

### Step 6: 写入 Milvus Lite

```
每条记录包含向量、文本、元数据
每 500 条批量插入
自动创建集合（如不存在）
```

**每条记录结构：**

| 字段     | 类型        | 示例                            |
| -------- | ----------- | ------------------------------- |
| vector   | FLOAT[1024] | [0.023, -0.045, ...]            |
| text     | VARCHAR     | "检查类型：CT\n部位：头颅\n..." |
| source   | VARCHAR     | "绥化*报告*模版2_row9.md"       |
| 检查类型 | VARCHAR     | "CT"                            |
| 部位     | VARCHAR     | "头颅"                          |
| 检查项目 | VARCHAR     | "颅脑平扫"                      |
| 诊断结论 | VARCHAR     | "脑出血"                        |

**对应代码：** `web.py` → `upload_and_process()` 中的插入逻辑

### Step 7: 更新元数据

```
调用 extract_metadata() 重新扫描 report_template/ 下所有 xlsx
全量重建 metadata.json（检查类型/部位/检查项目/诊断结论去重集合）
调用 reload_metadata() 刷新查询改写模块的模块级术语表
```

**为什么需要这一步？** 向量库和元数据是两套独立的系统：

| 系统          | 数据结构             | 更新方式                          | 用途                              |
| ------------- | -------------------- | --------------------------------- | --------------------------------- |
| Milvus 向量库 | 逐条记录的元数据字段 | 增量追加（按 source 去重）        | 检索时做元数据过滤和关键词匹配    |
| metadata.json | 全局去重的术语集合   | 全量重建（每次重新扫描所有 xlsx） | 查询改写时做术语标准化和 LLM 扩展 |

上传新 xlsx 后，Milvus 中的新记录可以立即被检索到，但 `metadata.json` 不会自动更新。如果不更新 metadata.json，查询改写模块就不知道新入库的检查类型、部位和诊断结论，可能导致：

- 新的部位别名无法被标准化（如新 xlsx 中出现了"骶骨"但 metadata.json 中没有）
- LLM 改写时无法扩展到新入库的检查项目或诊断结论

**全量重建 vs 追加：** `extract_metadata()` 每次都从头扫描所有 xlsx 文件，用 Python `set()` 去重后覆盖写入 metadata.json。虽然实现上是全量重建，但效果等价于追加——新文件的术语自然会被包含，已删除文件的术语也会自动消失。全量重建比增量追加更安全，不会因增量逻辑出错而遗漏或累积脏数据。

**对应代码：** `web.py` → `upload_and_process()` 中的元数据更新逻辑；`extract_metadata.py` → `extract_metadata()`；`query_rewrite.py` → `reload_metadata()`

## 二、报告生成流程

### Step 1: 用户输入 & 模糊检测

```
用户输入: "CT头部"
    │
    ▼
模糊检测（is_too_vague）
    │
    ├─ 过于模糊（如仅输入"CT"）→ 追问用户，终止流程
    │  返回追问提示，引导用户补充部位或诊断信息
    │
    ├─ 不够具体（如"CT头颅"，有类型+部位但无诊断）→ 进入改写流程
    │
    └─ 足够具体 → 直接使用原始查询
```

**模糊检测规则：**

| 输入示例   | 检查类型 | 部位 | 诊断 | 判定     | 处理     |
| ---------- | -------- | ---- | ---- | -------- | -------- |
| `CT`       | ✅       | ❌   | ❌   | 过于模糊 | 追问用户 |
| `MR`       | ✅       | ❌   | ❌   | 过于模糊 | 追问用户 |
| `CT头颅`   | ✅       | ✅   | ❌   | 不够具体 | 进入改写 |
| `CT脑出血` | ✅       | ✅   | ✅   | 足够具体 | 直接使用 |
| `脑出血`   | ❌       | ❌   | ✅   | 足够具体 | 直接使用 |

**对应代码：** `query_rewrite.py` → `is_too_vague()` + `get_clarification()` + `needs_rewrite()`

### Step 2: 术语标准化 & 查询改写

```
用户输入: "CT头部"
    │
    ▼
术语标准化（standardize_query）
    基于 metadata.json 中的标准部位名称
    "头部" → "头颅"（别名映射）
    "CT头部" → "CT头颅"
    │
    ▼
解析关键词（parse_query_keywords）
    从标准化文本提取结构化信息
    检查类型=CT, 部位=头颅, 诊断关键词=[]
    → 供路径2（元数据过滤）和路径3（关键词检索）使用
    │
    ▼
查询改写（rewrite_query）— 仅当 needs_rewrite=True 时
    基于 metadata.json 中的标准术语（含诊断结论，按检查类型+部位过滤）
    LLM 扩展为更完整的检索描述
    "CT头颅" → "CT头颅 颅脑平扫 颅脑平扫+增强"
    → 供路径1（向量检索）使用
```

**术语标准化原理：**

`metadata.json` 由 `extract_metadata.py` 从 xlsx 模板中提取，包含数据库中所有标准术语：

| 字段     | 示例值                       | 用途                  |
| -------- | ---------------------------- | --------------------- |
| 检查类型 | CT, MR, DR                   | 标准化检查类型        |
| 部位     | 头颅, 胸部, 腹部, ...        | 别名映射（头部→头颅） |
| 检查项目 | 颅脑平扫, 颅脑平扫+增强, ... | LLM 改写参考          |
| 诊断结论 | 脑出血, 脑梗死, ...          | LLM 改写参考          |

**改写约束：** LLM 只能使用 metadata.json 中存在的术语，不编造术语，不输出"影像学表现 诊断结论"等模板固定词。改写提示词注入四类元数据（检查类型、部位、检查项目、诊断结论），其中诊断结论按当前查询的检查类型+部位过滤，只注入相关子集，减少 token 消耗和噪声。

**关键词解析来源：** 从标准化文本解析（如 "CT头颅"），不从改写文本解析（如 "CT头颅 颅脑平扫"），确保路径2、路径3的关键词100%准确。

**对应代码：** `query_rewrite.py` → `standardize_query()` + `parse_query_keywords()` + `rewrite_query()` + `_build_rewrite_prompt()`；`extract_metadata.py`

### Step 3: 多路召回

术语标准化后，系统同时获得两种输入：**改写文本**（供向量检索）和**结构化关键词**（供元数据过滤和关键词检索），三路并行检索后合并去重：

```
标准化后:
  改写文本: "CT头颅 颅脑平扫 颅脑平扫+增强"
  关键词: {检查类型: "CT", 部位: "头颅", 诊断关键词: []}
    │
    ├──────────────────────────────────────────────────┐
    │                                                  │
    ▼                                                  ▼
路径1: 向量检索                              路径2: 元数据过滤
改写文本 → bge-m3 → 1024维查询向量           关键词:
    │                                         检查类型=CT, 部位=头颅
    ▼                                             │
Milvus 余弦相似度检索 → Top-K 候选               ▼
    │                                         Milvus 精确匹配查询
    │                                         检查类型=="CT" AND 部位=="头颅"
    │                                             │
    │                                             ▼
    │                                         返回匹配文档
    │                                                  │
    │                              ┌───────────────────┘
    │                              │
    │                              ▼
    │                     路径3: 关键词检索
    │                     关键词中的诊断关键词 → text like "%脑出血%"
    │                     AND 检查类型=="CT" AND 部位=="头颅"
    │                              │
    │                              ▼
    │                     返回全文匹配文档
    │                              │
    └──────────────────────────────┘
                   │
                   ▼
            合并去重（按 source 字段）
            记录每条文档的召回路径
                   │
                   ▼
              候选集（去重后）
```

**三路召回的输入来源：**

| 召回路径   | 输入来源     | 原理                              | 适用场景           |
| ---------- | ------------ | --------------------------------- | ------------------ |
| 向量检索   | 改写文本     | bge-m3 语义编码 + Milvus 余弦匹配 | 模糊查询、语义相似 |
| 元数据过滤 | 标准化关键词 | 检查类型/部位/诊断结论精确匹配    | 关键词明确的查询   |
| 关键词检索 | 标准化关键词 | Milvus like 全文匹配              | 专业术语检索       |

**为什么关键词从标准化文本解析？** 标准化文本（如 "CT头颅"）只包含用户原始意图的标准术语，解析结果100%准确。改写文本（如 "CT头颅 颅脑平扫 颅脑平扫+增强"）包含 LLM 扩展的术语，解析可能引入噪声。

**合并去重规则：** 按 `source` 字段去重，同一条文档被多路召回时记录所有召回路径（`_recall_paths`），`_recall_path` 标记为 `"multi"`。合并去重统一由 `multi_recall()` 函数处理，`web.py` 和 `chat.py` 均调用此函数，确保行为一致。

**过滤表达式安全：** 所有拼入 Milvus 过滤表达式的用户输入值均经过 `_esc_filter()` 转义，移除双引号和反斜杠，防止注入风险。

**多路召回 vs Rerank 对比：**

|          | 向量检索 (Bi-Encoder)                  | 元数据过滤                         | 关键词检索              | Rerank (Cross-Encoder)                      |
| -------- | -------------------------------------- | ---------------------------------- | ----------------------- | ------------------------------------------- |
| **输入** | 改写文本（LLM 扩展后）                 | 标准化关键词（从标准化文本解析）   | 标准化关键词            | 多路召回候选集                              |
| **模型** | bge-m3                                 | —（Milvus 精确匹配）               | —（Milvus like 查询）   | Qwen3-VL-Reranker-8B                        |
| **原理** | 问题和文档各自独立编码，计算余弦相似度 | 检查类型/部位/诊断结论字段精确匹配 | text 字段全文 like 匹配 | 问题和文档拼接输入，交叉注意力逐 token 比对 |
| **精度** | 粗筛，大致方向匹配                     | 精确匹配，零误判                   | 部分匹配，可能过宽      | 精排，逐字语义匹配                          |
| **速度** | 极快（向量预计算，只需算问题向量）     | 极快（索引查询）                   | 极快（索引查询）        | 较慢（每个文档都要完整过一遍模型）          |
| **适用** | 模糊查询、语义相似                     | 关键词明确的查询                   | 专业术语检索            | 从候选中精选最相关文档                      |

**对应代码：** `retrieval.py` → `multi_recall()` + `vector_search()` + `metadata_filter()` + `keyword_search()` + `_esc_filter()`；`query_rewrite.py` → `parse_query_keywords()`（关键词解析统一入口）

### Step 4: Rerank 精排

```
将多路召回的候选集文本和用户问题一起送入 Reranker
Cross-Encoder 逐 token 交叉比对
按 relevance_score 重新排序
取 Top-N（默认3条），每条带相关性分数
```

**Rerank 原理（Cross-Encoder）：**

```
                    ┌─────────────────────────┐
                    │   Qwen3-VL-Reranker-8B  │
                    │                         │
  "CT脑出血"  ──────►│  [CLS] CT 脑出血 [SEP]  │──────► relevance_score
                    │  检查类型：CT 诊断结论：  │        = 0.3328
  文档原文  ────────►│  脑出血 影像学表现：侧... │
                    │                         │
                    │   交叉注意力机制          │
                    │   每个token都能看到       │
                    │   问题和文档的所有token    │
                    └─────────────────────────┘
```

**Rerank 结果示例（top-3）：**

| Rerank 排名 | 原始位置 | 召回路径 | Rerank 分数 | 诊断结论           |
| ----------- | -------- | -------- | ----------- | ------------------ |
| 第1名       | 候选4    | vector   | 0.3328      | 脑出血             |
| 第2名       | 候选3    | multi    | 0.3279      | 脑出血             |
| 第3名       | 候选1    | metadata | 0.3193      | 脑出血（破入脑室） |

**为什么 Rerank 把候选4提到了第1名？**

- 候选1 "脑出血（破入脑室）" 虽然元数据精确匹配，但"破入脑室"是更具体的子类型
- 候选4 "脑出血" 与 "CT脑出血" 查询最精确匹配，且向量相似度也高
- Reranker 能理解这种细微差别，多路召回提供了更丰富的候选来源

**LLM 判断过程：**


## 多路召回 vs Rerank 对比

|          | 向量检索 (Bi-Encoder)                  | 元数据过滤                         | 关键词检索                             | Rerank (Cross-Encoder)                      |
| -------- | -------------------------------------- | ---------------------------------- | -------------------------------------- | ------------------------------------------- |
| **输入** | 改写文本（LLM 扩展后）                 | 标准化关键词（从标准化文本解析）   | 标准化关键词                           | 多路召回候选集                              |
| **模型** | bge-m3                                 | -                                  | -                                      | Qwen3-VL-Reranker-8B                        |
| **原理** | 问题和文档各自独立编码，计算余弦相似度 | 检查类型/部位/诊断结论字段精确匹配 | text 全文 like + 检查类型/部位精确匹配 | 问题和文档拼接输入，交叉注意力逐 token 比对 |
| **精度** | 粗筛，大致方向匹配                     | 精确匹配，关键词100%准确           | 部分匹配，可能漏掉语义相关             | 精排，逐字语义匹配                          |
| **速度** | 极快（向量预计算）                     | 极快（索引查询）                   | 快（全文索引）                         | 较慢（每对实时计算）                        |
| **适用** | 从大量数据中快速召回候选               | 关键词明确的查询                   | 专业术语检索                           | 从候选中精选最相关文档                      |

**设计逻辑：** 多路召回（向量检索 + 元数据过滤 + 关键词检索）负责从不同角度"广撒网"捞回候选，合并去重后交给 Rerank "慢而精"地从候选中选出最相关的给 LLM。

> 📖 更多 Rerank 原理、LLM 判断逻辑、参数含义、降级机制详见 [Rerank.md](Rerank.md)
